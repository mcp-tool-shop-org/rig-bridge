# v1.1 Roadmap

> **Status:** Forward plan. Single review surface for v1.1 dispatch.

v1.0.0 ships the git transport as a complete standalone product. v1.1 lands the cross-system features that the Phase 0 design specified but that v1.0.0 deliberately deferred.

## v1.1 Trigger — coupling-driven, not calendar-driven

v1.1 dispatches when **testing-os next ships a schema bump for any reason**. That's the natural moment to land the `bridge_messages` + `bridge_message_events` tables alongside whatever else triggered the bump. rig-bridge v1.1 then dispatches in the same coordinated window.

Until that trigger fires, v1.0.0's git transport carries the load. There is no calendar-bound v1.1 release date.

## v1.1 Scope — 4 items in one coherent dispatch

### 1. Control-plane integration (the writes-through path)

Implement what `docs/control-plane-integration.md` specifies. Two halves:

**testing-os half** (separate swarm in `mcp-tool-shop-org/dogfood-lab/testing-os`):
- Add `bridge_messages` + `bridge_message_events` tables in `packages/dogfood-swarm/db/schema.js`
- Bump `SCHEMA_VERSION` from 4 to 5 (or whatever the next bump value is when v1.1 fires)
- Add migration runner that creates the tables on existing v4 DBs
- Add 3 indexes per the §2.2 design
- Add the workspace export `@dogfood-lab/dogfood-swarm/bridge` (subpath export per testing-os CLAUDE.md §4)

**rig-bridge half** (this repo):
- `src/engine/control-plane.ts` — the writes-through module, calling out via the `@dogfood-lab/dogfood-swarm/bridge` export
- `bridge sync --check` command — pre-wave hook for cross-rig coherence (per `docs/control-plane-integration.md` §3.3)
- `bridge replay` command — reconstruct CP state from git history (per §5.3)
- All 4 v1.0.0 commands gain a CP write step at the appropriate point in their sequences

Both halves ship together. Neither is useful alone.

### 2. `tldr` validator (parked Phase 0 question Q1)

`docs/envelope-spec.md` §3.2 declares `tldr` as ≤280 chars "soft cap, validator only" but no validator exists in v1.0.0. v1.1 decision required:

- **Option A**: `bridge send` enforces the cap, refusing to send if exceeded
- **Option B**: `bridge send` warns (stderr) but proceeds
- **Option C**: drop the cap from the spec entirely

Pick one in v1.1. Document the choice with reasoning.

### 3. Multi-recipient envelope row count (parked Phase 0 question Q2)

`docs/control-plane-integration.md` §4 cross-walk asserts "multi-recipient envelopes land as one row per recipient at the courier seam." Alternative: one row with `to_rig` as a JSON array.

The choice has indexing + query-shape consequences:
- Per-recipient rows: simple SQL queries (`WHERE to_rig = ?`), but a 5-recipient envelope produces 5 rows
- JSON array: one row per envelope, but `to_rig LIKE '%foo%'` queries are fragile

v1.1 implementor's call when the CP code lands. Document the choice with reasoning.

### 4. `display_name` Unicode handling (parked Phase 0 question Q3)

`schemas/bridge-message.schema.json` enforces `display_name` ≤80 chars but no character-set restriction (Unicode allowed). v1.1 review:

- Does any rendering surface (terminal output, GitHub markdown, control-plane queries) break on non-ASCII display names?
- If yes, restrict the schema. If no, leave open (the field is metadata-only).

This is review, not necessarily a code change.

## Out of scope for v1.1

- Multi-rig (3+) topology — symmetric two-rig is still the v1.x model
- MCP frontend — `bridge` is CLI-only through v1.x
- Cloud-hosted central queue — git remains the wire
- Encryption beyond GitHub TLS — repo permissions are the auth model

These remain out of scope per `docs/envelope-spec.md` §7 and `ARCHITECTURE.md` "Out of scope for v1.0.0". v1.1 inherits all v1.0.0 deferrals.

## v1.0.0 boundary — what v1.0.0 ships

For clarity:
- `bridge init` — initialize local clone, install commit-message hook, write `.bridge/config.yaml`
- `bridge new <thread-id>` — scaffold from template
- `bridge send <type> --thread <id>` — validate + body_hash + marker→class derivation + canonical rig-id ingress + git commit + push (NO control-plane write)
- `bridge close <thread-id>` — write RESOLUTION.md + commit + push (NO control-plane write)
- Engine helpers: envelope parse/render, schema validation, body_hash with §4.1 normalization, status_class derivation, rig-id validation, git wrapper with submodule + size guards

That's the complete v1.0.0 product surface. Phase 7 of the dogfood swarm extends it (`status`, `thread`, `sync`, `relay` + engine library extraction). Phase 8 adds guards. Phase 10 ships v1.0.0 via Full Treatment.

v1.1 begins the moment testing-os ships its next schema bump.
